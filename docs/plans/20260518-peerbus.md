# peerbus — agent-agnostic message bus + adapters

## Overview

`peerbus` is a small open-source Go project: an **agent-agnostic message bus** plus thin **adapters** that let heterogeneous AI agents send and broadcast messages to each other through one broker.

- **Problem it solves:** today, Claude-Code-to-Claude-Code messaging exists only via `cc2cc` (single-machine, filesystem, Claude-only, per-session `server.mjs` that orphans and burns CPU). There is no clean, durable, cross-machine, *heterogeneous* bus that also lets a non-Claude agent (Hermes, OpenClaw, Codex CLI, …) participate. Driving Claude Code programmatically (`claude -p` / Claude Agent SDK) is explicitly off the table for pricing/policy reasons, so the integration must be **out-of-band peer messaging between already-running interactive agents**, not one agent driving another.
- **Key benefits:** one bus subsumes `cc2cc` entirely (all Claude sessions + other agents on one fabric); durable at-least-once delivery; cross-machine; role-neutral transport; both sides replaceable via adapters; Claude stays a normal interactive tmux subscription session.
- **Integration:** Claude Code participates via its official **channels** feature (an MCP server with the `claude/channel` capability, push-waking an idle session). Every other agent participates via a generic MCP adapter it drains on its own schedule. The "only escalate to the human when a real decision is needed" policy lives entirely in the *consuming agent's* prompt (e.g. Hermes SOUL/skill), **never** in peerbus.

**Honest taxonomy:** this is a custom MCP-channel peer bus. It is *conceptually* A2A-shaped (peer agents, async messages, human escalation handled by the peer) but is **not** an implementation of Zed's Agent Client Protocol nor Google/LF Agent2Agent. The README must state this plainly.

**Inspired by / credit (required in README):**
- [`non4me/cc2cc`](https://github.com/non4me/cc2cc) — the direct inspiration; peerbus is its generalized successor (link its README).
- [`louislva/claude-peers-mcp`](https://github.com/louislva/claude-peers-mcp) — prior art for broker-backed Claude peer messaging over channels.

## Solution Overview

**Topology:** one long-lived **broker** service (operated as a managed compose/s6 service, *never* per-session-spawned) + many thin, mostly-ephemeral **adapter** processes whose lifecycle is owned by each agent runtime.

- **Broker:** Go, WebSocket, 100% agent-agnostic (zero per-agent code). Durable SQLite queue + hash-chain audit. Verbs: `register`, `send`, `deliver`, `ack`, `peers`. Per-connection bearer-token auth; per-message HMAC carried end-to-end and verified by recipients (a compromised broker cannot forge a peer).
- **Delivery:** at-least-once, durable, dedupe-by-message-id, per-sender FIFO (no global order). Addressing: auto-registered unique peer name; `to:<name>` direct, `to:*` broadcast only. Offline recipients queued in SQLite, delivered on reconnect/next drain. Unacked redeliver on reconnect (consumers dedupe). Broadcast fans out to *currently-registered* peers except sender (each its own durable copy + own ack); **no backfill** for not-yet-registered peers.
- **Adapters:** one Go binary, `--adapter=<mode>`, any number of instances, additive future modes; broker never knows the mode.
  - `--adapter=cc`: *is* the MCP `claude/channel` server, spawned per Claude session by Claude Code over stdio. Inbound = channel push-wake; outbound = MCP tools `bus.send` / `bus.broadcast` / `bus.peers`. N sessions ⇒ N short-lived adapters, each a distinct peer (fixes the orphan bug — lifecycle bound to the session, broker outlives all).
  - `--adapter=generic`: plain stdio MCP server, spawned per drain-agent. Tools: `bus.send` / `bus.broadcast` / `bus.peers` / `bus.drain`. No push; host agent calls `bus.drain` on its own schedule.

**Key design decisions & rationale:**
- *Broker/adapter split* — broker must outlive sessions to hold the durable queue; cc adapter must die with its stdio session. Conflating them is exactly cc2cc's orphaned-`server.mjs` bug; the split designs it out.
- *Role-neutral bus* — escalation/"think first, only ping the human on a real decision" is a property of the consuming agent's prompt, keyed off a `source: peer-bus` tag carried in the message envelope. peerbus ships the generic adapter and a *documented* Hermes drain-skill pattern, never Hermes prompts.
- *HMAC end-to-end* — full cc2cc parity; integrity independent of broker trust.

## Technical Details

- **Module:** `go.mod` module path placeholder `github.com/OWNER/peerbus` — confirm the real GitHub slug at first push (name "peerbus" cleared of major collisions; npm `peerbus` free; verify the org/repo slug at repo creation).
- **Message envelope (`internal/wire`):** `protocol_version`, `id` (ULID/uuidv7), `from`, `to` (name or `*`), `ts`, `source` (e.g. `peer-bus`), `kind` (`msg`/`broadcast`), `body` (`json.RawMessage` — opaque, hashed verbatim, never re-encoded), `hmac` (hex). Newline-delimited JSON over the WS data channel; control frames (`register`/`ack`/`peers`) are typed JSON-RPC-ish objects.
- **Protocol version policy (v1):** single `protocol_version` field, **exact-match-or-reject** at `register` (no negotiation engine — the field exists so future negotiation is additive, but v1 does not implement it).
- **HMAC canonicalization (load-bearing — the project's primary correctness property):** the signed bytes are produced by marshalling a **fixed-field-order canonical subset struct** that omits `hmac`, with `body` carried as `json.RawMessage` and hashed **verbatim** (never re-marshalled — re-encoding opaque JSON is not byte-stable). Sender signs the canonical form; recipient reconstructs the canonical form from the *received wire bytes* (marshal→unmarshal→canonicalize→verify) — the two must agree across machines and `protocol_version` values. `encoding/json` struct output with a fixed field order and no omitempty on signed fields is the canonical form; this is asserted by cross-marshal and cross-version test vectors, not a same-process round-trip.
- **Auth:** static bearer token(s) from broker config/env; a peer name is bindable only under a valid token; duplicate-name claim under the same token = takeover (old conn closed), under a different token = reject.
- **HMAC:** HMAC-SHA256 over canonical envelope (excluding `hmac`), shared secret distributed to peers out-of-band (config/env). Recipient verifies before surfacing.
- **Store (`internal/store`, SQLite, `modernc.org/sqlite` pure-Go, WAL):** tables `peers`, `messages` (with `delivered`, `acked`, `attempts`, `seq` per-sender), `audit`. Dedupe by `id` unique constraint. Per-sender FIFO via monotonic `seq`. Offline = rows with `delivered=0` for a recipient name, flushed on register/drain.
- **Audit (`internal/audit`):** append-only `audit` rows, each `hash = blake3(prev_hash || canonical_event)`; genesis `blake3("")`; `peerbus-broker audit verify` walks and reports breaks.
- **Channel adapter (`internal/channel`):** implements the MCP `claude/channel` capability handshake; maps broker `deliver` → `notifications/claude/channel` (wakes session); registers `bus.*` MCP tools mapping to broker `send`/`peers`.
- **Generic adapter (`internal/mcp`):** stdio MCP server exposing `bus.send`/`bus.broadcast`/`bus.peers`/`bus.drain`; `bus.drain` returns + acks pending messages.

**⚠️ ACCEPTANCE DEVIATION — broadcast HMAC is broker-trusted, NOT end-to-end (KNOWN, surfaced honestly):**
The Solution Overview claims the design property *"HMAC end-to-end — full cc2cc parity; integrity independent of broker trust."* This holds for **direct messages** (`TestParity_HMACSignedDeliversAndForgedRejected` proves a forged-secret direct message is rejected and a correctly-signed one verifies end-to-end). It does **NOT** hold for **broadcast** (`to:*`): `internal/broker/router.go` `routeBroadcast` rewrites the signed fields per recipient — `copyEnv.ID = env.ID + "|" + r` and `copyEnv.To = r` — before re-marshalling each per-recipient copy to the wire. Because the recipient reconstructs the canonical form from the *received wire bytes* (which now carry the broker-rewritten `id`/`to`), the original sender's HMAC no longer verifies on broadcast copies. Adapters log-and-skip these (`internal/adapter/generic.go`: broadcast copies "fail HMAC at Recv and the resume loop logs+skips them"); `TestParity_BroadcastFanOutSenderExcluded` *tolerates* `"failed HMAC verify"` as a valid outcome rather than asserting integrity. **Conclusion: broadcast-message HMAC integrity is broker-trusted, not end-to-end** — a compromised broker can tamper with broadcast copies undetected, contradicting the stated end-to-end property for the broadcast case. Direct messages remain genuinely end-to-end HMAC-verified. This is a real deviation, NOT fixed here (Task 14 is verification only; already flagged for review phase 1 / tracked in Task 10 & `internal/adapter/generic.go`). Acceptance is granted **with this recorded deviation**: the project does NOT deliver full end-to-end HMAC for broadcast as the Solution Overview asserts; it delivers end-to-end HMAC for direct messages and broker-trusted integrity for broadcast fan-out.
